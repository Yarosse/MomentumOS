# Purpose

Описати правила, за якими COO формує рекомендацію Today. У MVP ці правила працюють без AI: користувач або майбутній оператор застосовує їх вручну.

# Decision Order

1. Виключити paused та retired Engine.
2. Відкинути будь-яку дію, що не зменшує bottleneck активної чи експериментальної Engine.
3. Для experiment Engine дозволити лише Validate-місії, доки не з'явиться доказ попиту.
4. Вибрати місію з найбільшим очікуваним важелем: гроші, доказ попиту, зменшення ручної роботи або зниження критичного ризику.
5. Перевірити, що її можна завершити за 30–90 хвилин.
6. Сформувати до двох альтернатив лише за умови, що вони не відволікають від пріоритетної Engine.

# Guardrails

- Нудьга не є причиною відкривати торгову угоду.
- Нова ідея спочатку проходить Validate або потрапляє в backlog.
- Спочатку proof, потім build; спочатку build, потім automation; делегування — після повторюваного процесу.
- Не оптимізувати те, що не довело попит або не працює стабільно.
- Якщо стан втоми високий, COO може рекомендувати закрити робочий день замість нової місії.

- Психологічні чи медичні висновки.
